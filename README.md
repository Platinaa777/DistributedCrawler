# Distributed Crawling System (Go)

Распределённая система краулинга: задачи на обход/парсинг публикуются в RabbitMQ, далее координаторр распределяет их по воркерам, а воркеры ходят в интернет, извлекают данные и сохраняют результаты.

> Репозиторий ориентирован на реализацию на **Go**. Архитектура и роли компонентов соответствуют описанию в документации проекта.

## Зачем это нужно

- Масштабируемый сбор веб‑данных (параллельно на нескольких воркерах).
- Централизованное управление задачами и мониторинг статусов.
- Нормализация и сохранение результатов (например, в PostgreSQL), плюс хранение «сырых» страниц (например, в S3/MinIO).

## Архитектура (в двух словах)

```
Producer -> RabbitMQ -> Coordinator -> Workers -> Internet
                         |     |
                         |     +--> Results/Raw pages -> Storage (Postgres / S3)
                         +--> UI / API (управление и мониторинг)
```

- **Coordinator** — центральный сервис: читает/планирует задачи, распределяет работу, следит за статусами, логами и трассировками.
- **Workers** — исполняемые процессы: получают задания, скачивают страницы, парсят, нормализуют и сохраняют результаты.
- **RabbitMQ (RMQ)** — транспорт для постановки задач и доставки их воркерам.
- **Storage** — БД для нормализованных данных + blob‑хранилище для сырых HTML/JSON.

## Базовые use cases

### 1) Одноразовый парсинг списка URL

1. Вы формируете сообщение-задачу (список URL, глубина, фильтры, «шаблон» извлечения).
2. Публикуете сообщение в очередь RMQ.
3. Координатор распределяет задачу на воркера.
4. Воркер скачивает страницы, извлекает данные, сохраняет результат и обновляет статус.

### 2) Регулярный мониторинг (по расписанию)

1. Публикуете задачи с параметром расписания/частоты (или используете внешний планировщик, который пушит сообщения).
2. Координатор ставит задачи в очередь в нужное время.
3. Воркеры обрабатывают и обновляют результаты (например, для цен/контента/SEO‑метрик).

### 3) Краулинг с обходом ссылок (depth + фильтры)

1. В задаче указываете начальные URL и `depth`.
2. Воркеры дополнительно извлекают ссылки со страниц, применяют правила фильтрации и продолжают обход.
3. Сырые страницы сохраняются отдельно, нормализованные данные — в БД.

### 4) Извлечение структурированных данных из HTML/JSON

1. В задаче указываете «extractor» (например, CSS/XPath селекторы или иной шаблон).
2. Воркеры вытаскивают нужные поля, приводят к единому формату (даты/цены/числа) и сохраняют.

## Очереди и формат сообщений (пример)

Ниже — **пример** payload для RMQ (формат можно адаптировать под ваш домен и схему хранения):

```json
{
  "task_id": "uuid",
  "type": "crawl_and_parse",
  "urls": ["https://example.com"],
  "depth": 1,
  "filters": {
    "allow_domains": ["example.com"],
    "deny_patterns": ["logout", "cart"]
  },
  "extractor": {
    "kind": "css",
    "fields": {
      "title": "h1",
      "price": ".price"
    }
  },
  "output": {
    "save_raw": true,
    "save_normalized": true
  },
  "meta": {
    "priority": 5,
    "scheduled_at": "2026-01-01T00:00:00Z"
  }
}
```

Рекомендуемые практики:
- Делайте сообщения **идемпотентными**: `task_id` + детерминированный ключ результата.
- Поддерживайте **retry/backoff**: NACK/повторная постановка в очередь при временных ошибках.
- Храните **статусы задач**: `queued → running → done/error`.

## Запуск (типовой сценарий)

> Команды ниже — шаблон. Подстройте под структуру вашего репозитория (`./cmd/...`, Makefile и т.п.).

### Зависимости

- Go (рекомендуется актуальная LTS версия)
- RabbitMQ
- PostgreSQL (для нормализованных данных)
- MinIO/S3 (для сырых страниц) — опционально

Если у вас есть `docker-compose.yml`:

```bash
docker compose up -d rabbitmq postgres minio
```

### Конфигурация (env)

```bash
export RABBITMQ_URL="amqp://guest:guest@localhost:5672/"
export POSTGRES_DSN="postgres://user:pass@localhost:5432/db?sslmode=disable"
export S3_ENDPOINT="http://localhost:9000"
export S3_ACCESS_KEY="minioadmin"
export S3_SECRET_KEY="minioadmin"
```

### Запуск координатора и воркера

```bash
go run ./cmd/coordinator
go run ./cmd/worker
```

### Публикация задачи в RMQ

Варианты:
- любой ваш продьюсер (CLI/скрипт/сервис) публикует JSON в нужную очередь;
- или простой утилитный `cmd/producer` (если есть в репозитории).
