# Example 11

Этот пример показывает краулинг личной ленты Habr с аутентификацией через Cookie.

## Что лежит в папке

- `request11.json` — конфигурация задания с cookie-аутентификацией.
- `response11.json` — пример результата выполнения.

## Что показывает запрос

`request11.json` описывает сценарий, в котором crawler:

- стартует с `https://habr.com/ru/feed/` — персональная лента, доступная только авторизованным пользователям;
- передаёт куку `habr_session=...` в заголовке `Cookie` каждого запроса;
- работает в режиме `CRAWL_MODE_PAGINATION_ONLY` — ходит по страницам пагинации ленты;
- ограничен доменом `habr.com`;
- из каждой карточки статьи извлекает `title`, `url`, `author`, `hubs`, `views`.

## Как работает cookie-аутентификация

Поле `auth.cookie` принимает строку в формате стандартного HTTP Cookie-заголовка:

```
"auth": {
  "cookie": "habr_session=abc123def456; _ga=GA1.2.111111111.1700000000; hl=ru"
}
```

Fetcher подставляет эту строку как есть в заголовок `Cookie: <value>` каждого HTTP-запроса. Если используется browser-режим (chromedp/selenium), куки разбиваются на пары `name=value` и добавляются в сессию браузера поштучно через WebDriver API.

## Как получить куку для своего сайта

1. Откройте DevTools браузера (F12) → вкладка **Application** → **Cookies**;
2. Скопируйте нужные куки, отвечающие за сессию (обычно `session_id`, `auth_token`, `PHPSESSID` и т.п.);
3. Склейте их в одну строку через `; ` и подставьте в `auth.cookie`.

**Важно:** куки имеют срок жизни. Для scheduled-заданий (`job_type: JOB_TYPE_SCHEDULED`) может потребоваться периодическое обновление конфигурации.

## Что важно для спеки

- `auth.cookie` — единственное заполненное auth-поле; остальные (`basic_user`, `basic_password`, `bearer_token`) оставлены пустыми.
- Если сайт возвращает `302 → /login` для неавторизованных запросов, страницы будут распарсены, но поля вернут `null` — это сигнал о том, что куки устарели.
- Пагинация habr-ленты идёт через `a[id='next_page']`.
