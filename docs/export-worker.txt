Goal

ParserWorker must persist extraction output to S3 (instead of printing), and store the result file URL (or object key) in DB on the corresponding crawl_task.

Implement a new worker ExportWorker (same style as FetchWorker/ParserWorker) that:

finds crawl_jobs that are fully finished

collects results for all tasks in the job from S3

builds a final report JSON and/or CSV

uploads final report to S3

stores final report URL in DB (job-level preferred), and marks job/export status to avoid re-exporting.

Requirements

Replace printResults() with something like persistResults():

Marshal output to JSON (same structure as current console output).

Upload JSON bytes to S3 under deterministic path:

Part B — ExportWorker
Purpose

Produce job-level export file(s) after all tasks for a job are terminal (Completed or Failed), using already stored per-task result objects in S3.

“Job is fully finished” definition

A job qualifies for export when:

it has no tasks in non-terminal states (e.g., Pending/Running/etc.)

i.e., every task is either Completed or Failed.

Behavior

Periodically poll DB for eligible jobs that are not yet exported.

For each eligible job:

List tasks of the job from DB.

For each task with TaskStatusCompleted:

load task result JSON from S3 using result_object_key (or URL).

aggregate into a report structure.

For failed tasks:

include minimal info in report (task_id, url, error/reason if stored).

Generate:

report.json (required)

report.csv (optional but requested; implement if feasible now)

Upload report file(s) to S3:

Example: exports/jobs/{job_id}/report.json, exports/jobs/{job_id}/report.csv

Store export references in DB:

Add to crawl_jobs columns like export_json_key, export_csv_key, exported_at, export_status

or create separate table crawl_job_exports (ok for MVP to store on job).

Mark exported so worker does not repeat work.

Idempotency & locking (MVP-level)

Add job field export_status with states: NOT_STARTED, IN_PROGRESS, COMPLETED, FAILED.

ExportWorker should atomically transition NOT_STARTED -> IN_PROGRESS before doing work (transaction/compare-and-swap style).

If already COMPLETED, skip.

If IN_PROGRESS is stale (optional), ignore for MVP.

CSV format (if implemented)

Each row represents one task result flattened:

Columns: task_id, url, plus field keys (dynamic)

If fields vary across tasks:

Compute union of all field names across completed tasks for header.

Missing values empty.

Metrics can be separate file or embedded as JSON string columns (choose simplest).

status transition methods

aggregation + JSON/CSV generation
