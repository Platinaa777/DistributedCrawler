You are implementing simple authentication/authorization for an MVP backend in Go.

Context / Goals

Implement JWT-based auth with:

Access token: short-lived, minimal claims

Refresh token: long-lived, stored server-side (DB) and used to rotate tokens

Add a users table with:

email (unique)

password_hash (bcrypt or argon2)

Integrate with existing HTTP API (grpc-gateway / REST). Keep it minimal and production-ish.

Tech Constraints

Language: Go

Password hashing: bcrypt (preferred for MVP)

JWT library: github.com/golang-jwt/jwt/v5

Database: PostgreSQL (migrations via goose)

Do not implement complex features (OAuth, SSO, roles system) unless required below.

Security baseline: prevent obvious footguns (plaintext passwords, unlimited refresh reuse).

Requirements
1) Database Schema (Postgres + goose migrations)

Create tables:

users

id UUID or VARCHAR(36) (your project uses valueobjects.ID — align with it)

email VARCHAR(255) UNIQUE NOT NULL

password_hash TEXT NOT NULL

created_at TIMESTAMPTZ NOT NULL DEFAULT now()

updated_at TIMESTAMPTZ NOT NULL DEFAULT now()

refresh_tokens

Store refresh token as hash (never plaintext), tied to a user and revocable.

id UUID / VARCHAR(36) PRIMARY KEY

user_id FK -> users(id) ON DELETE CASCADE

token_hash TEXT NOT NULL UNIQUE

expires_at TIMESTAMPTZ NOT NULL

revoked_at TIMESTAMPTZ NULL

created_at TIMESTAMPTZ NOT NULL DEFAULT now()

Optional (nice): replaced_by_token_id to support rotation chain (nullable)

Indexes:

refresh_tokens(user_id)

refresh_tokens(token_hash) unique already

2) Token Design
Access Token (JWT)

Signed with HS256 using JWT_SECRET env var.

TTL: 10–15 minutes (MVP default: 15m)

Minimal claims:

sub: user id

exp, iat

optionally email (only if really needed; prefer not)

Audience/issuer optional for MVP; if added:

iss: service name

aud: “api”

Refresh Token

Random opaque string (at least 32 bytes) generated via crypto/rand.

TTL: 7–30 days (MVP default: 30d)

Store only SHA-256 hash of refresh token in DB:

token_hash = hex(sha256(refreshToken))

On refresh:

Validate token exists, not revoked, not expired

Rotate: revoke old token, create new refresh token row, return new pair

3) API Endpoints (HTTP, grpc-gateway style)

Implement minimal endpoints:

POST /api/v1/auth/register

Body: { "email": "...", "password": "..." }

Validate email format, password min length (>= 8)

Hash password, insert user

Return 201 and optionally auto-login tokens (OK for MVP). If you do, return token pair.

POST /api/v1/auth/login

Body: { "email": "...", "password": "..." }

Verify user exists and password matches hash

Issue access + refresh tokens

Return: { "access_token": "...", "refresh_token": "...", "expires_in": 900 }

POST /api/v1/auth/refresh

Body: { "refresh_token": "..." }

Rotate refresh token and return new pair

POST /api/v1/auth/logout

Body: { "refresh_token": "..." } OR logout-all

Revoke that refresh token in DB (set revoked_at = now()).

Return 204.

4) Auth Middleware

Protect all endpoints except:

/api/v1/auth/register

/api/v1/auth/login

/api/v1/auth/refresh

Middleware behavior:

Read Authorization: Bearer <access_token>

Validate JWT signature + exp

Extract sub (user_id)

Put user_id into request context (and for gRPC gateway, into context.Context)

Return 401 if missing/invalid/expired token

Also ensure CORS allows Authorization header (if you have browser UI).

5) Implementation Structure (suggested)

Create package/module layout:

internal/auth

password.go (hash/compare)

jwt.go (sign/verify)

refresh.go (generate refresh token, hash it)

middleware.go (for grpc-middleware to extract it and pass to http server)

internal/domain/users (or equivalent)

repo.go (CRUD for users + refresh tokens)

service.go (Register/Login/Refresh/Logout)

internal/api/http or your existing API layer

handlers for endpoints above

Use dependency injection consistent with existing code (serviceProvider pattern).

6) Environment / Config

Add config keys (env-based):

JWT_SECRET (required)

ACCESS_TOKEN_TTL (default 15m)

REFRESH_TOKEN_TTL (default 720h / 30d)

Fail fast if JWT_SECRET missing.

7) Security Notes (MVP level)

Passwords: bcrypt cost 10–12 (pick 12 if ok)

Refresh token:

store only hash

rotate on every refresh

revoke on logout

Rate limit login attempts is optional (nice-to-have; can be omitted)

Use constant-time compare for token hash checks if implementing manually (DB lookup by hash is OK)

8) Deliverables

Implement end-to-end working auth:

Goose migrations for new tables

Repo + service logic

HTTP handlers for auth endpoints

Middleware protecting API routes

Basic unit tests for:

password hashing/compare

JWT sign/verify

refresh rotation logic (happy path + revoked/expired)

Update OpenAPI / swagger annotations if your grpc-gateway exposes them.

9) Definition of Done

I can register a user, login, call a protected endpoint with access token, refresh when access expires, and logout invalidates refresh token.

Refresh token rotation works: old refresh cannot be used twice.

No plaintext refresh tokens stored in DB.

Passwords stored only as hash.
