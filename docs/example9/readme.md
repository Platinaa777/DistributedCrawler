# Example 9

Этот пример показывает постраничный обход результатов поиска в Wikipedia без перехода в сами статьи.

## Что лежит в папке

- `request9.json` - конфигурация одноразового задания.
- `readme.md` - описание сценария и того, как его повторить.

## Что показывает запрос

`request9.json` описывает сценарий, в котором crawler:

- стартует со страницы `Special:Search` на `en.wikipedia.org`;
- использует запрос `programming language history`, который на 9 апреля 2026 года открывается именно как search results, а не как редирект в одну статью;
- работает в режиме `CRAWL_MODE_PAGINATION_ONLY`;
- идет по страницам выдачи через селектор `a[rel="next"]`;
- собирает page-level поле `page_title`;
- на каждой странице результатов извлекает items:
  - `title`;
  - `url`;
  - `snippet`;
  - `meta`;
  - `thumbnail`.

Это Wikipedia-аналог Example 1: есть одна listing-страница, по которой нужно идти только по пагинации и собирать карточки найденных элементов.

## Почему это хороший шаблон

- Его легко повторить: меняешь только поисковую фразу в seed URL.
- Он хорошо подходит для research-сценариев, где нужен тематический набор статей без захода в каждую статью.
- Селекторы здесь стабильнее и чище, чем на произвольных list-страницах Wikipedia.

## Что генерируется в ответ

Ожидаемая форма результата такая же, как в `example1`:

- верхний уровень: `job_id`, `exported_at`, `results`, `total_tasks`;
- каждый `result` соответствует одной странице поисковой выдачи;
- `fields.page_title` хранит заголовок страницы;
- `items` содержит массив найденных search results.

## Что важно для спеки

- Для этого сценария критично оставлять `fulltext=1`, иначе Wikipedia может сделать редирект на точное совпадение и поисковой выдачи не будет.
- При `CRAWL_MODE_PAGINATION_ONLY` полезные данные лежат в `items`, а не в page-level `fields`.
- Это хороший reference для паттерна "поисковая выдача + пагинация + без перехода в detail page".
