Полный текст программы расположен в публичном GitHub-репозитории проекта distributed-crawler.

Платформа представляет собой распределённую систему веб-краулинга, реализованную на языке Go. Система
состоит из нескольких независимых сервисов (бинарных файлов), каждый из которых располагается в
собственной поддиректории внутри cmd/. Инфраструктура поднимается с помощью Docker Compose — файл
docker/docker-compose.yaml запускает PostgreSQL, RabbitMQ, MinIO и Redis. Для развёртывания на Kubernetes
используется Helm-чарт, расположенный в deploy/helm/distributed-crawler/.

Каждый сервис имеет единую структуру, опирающуюся на принципы чистой архитектуры. Состав
основных директорий следующий:

    cmd/                 — точки входа в программу (по одной на каждый сервис)
    api/v1/              — proto-файлы с описанием API (buf.yaml, buf.gen.yaml)
    pkg/v1/              — сгенерированный на основе proto Go-код (пакет crawlergrpc)
    internal/            — основная директория с кодом всех сервисов
    deploy/helm/         — Helm-чарт для развёртывания на Kubernetes
    docker/              — файлы инфраструктуры (docker-compose.yaml)
    observability/       — конфигурации Grafana, OpenTelemetry


1. Бинарные файлы (cmd/)

grpc_server — основной API-сервер. Запускает одновременно: gRPC-сервер (порт 8083),
HTTP-шлюз на основе grpc-gateway (порт 8084), фоновую горутину публикации outbox-событий
и горутину планировщика заданий. Все четыре компонента работают в одном процессе и
управляются через internal/app/api_app.go.

fetch_worker — воркер-загрузчик. Читает задачи из очереди crawl_queue, загружает HTML или
рендерит страницу через headless-браузер (chromedp), сохраняет снапшот в MinIO, публикует
задачу в очередь parsing_queue.

parser_worker — воркер-парсер. Читает задачи из parsing_queue, извлекает данные по правилам
ExtractionSpec (CSS-селекторы, атрибуты, трансформации), сохраняет ExtractedRecord в базу
данных, обнаруживает новые ссылки и создаёт новые задачи краулинга.

export_worker — воркер экспорта. Опрашивает завершённые задания, формирует файлы экспорта
в форматах JSON и CSV, загружает их в MinIO, обновляет статус задания.

scheduler_worker — воркер планировщика. Опрашивает расписания (cron-выражения) и создаёт
новые циклы краулинга для заданий типа JOB_TYPE_SCHEDULED.

memory_broker — автономный gRPC-сервис очередей, реализованный в памяти. Используется как
альтернатива RabbitMQ/Kafka для локальной разработки и тестирования.

migrate — утилита для применения миграций базы данных (Goose).

grpc_service — вспомогательный gRPC-клиент для ручного вызова методов API.


2. Слои архитектуры (internal/)

2.1 Доменный слой (internal/domain/)

Содержит бизнес-сущности и интерфейсы репозиториев без зависимостей на инфраструктуру.

    domain/crawl/models/         — основные модели: CrawlJob, CrawlJobConfig, CrawlTask,
                                   ExtractionSpec, ScopeRules, RateLimitPolicy, RetryPolicy,
                                   AuthOptions, ScheduleOptions, OutboxEvent, Preview
    domain/crawl/valueobjects/   — типизированные идентификаторы: CrawlJobID, CrawlTaskID,
                                   PageSnapshotID, ExtractedRecordID и другие (UUID)
    domain/crawl/repos/          — интерфейсы репозиториев (CrawlJobRepo, CrawlTaskRepo и др.)
    domain/crawl/events/         — доменные события (TaskEnqueuedEvent и др.)
    domain/crawl/services/       — доменные сервисы
    domain/auth/                 — модели пользователей, refresh-токенов, ролей
    domain/queue/                — модели и интерфейсы для управления очередями (QueueEndpoint)
    domain/shared/               — общие ошибки (errors.go), абстракция Clock

Модель CrawlJob хранит идентификатор конфигурации, статус, временные метки создания и
завершения, а также ключи S3-объектов для JSON/CSV-экспорта. Модель CrawlJobConfig
описывает правила краулинга: начальные URL (Seeds), правила области обхода (ScopeRules),
спецификацию извлечения данных (ExtractionSpec), политику rate-limit, политику повторных
попыток, параметры аутентификации на целевом сайте, расписание (cron), тип задания и режим
обхода ссылок (CrawlMode).

ExtractionSpec задаёт список полей (FieldSpec) с CSS-селекторами, типами значений (string,
int, float, bool, url, json), трансформациями (trim, lower, normalize_url, parse_price,
html_to_text, sha256 и другие) и правила постраничного обхода (PaginationSpec). Также
поддерживается структурированное извлечение по контейнеру (ItemsSpec) — когда каждый
элемент списка на странице становится отдельной записью.

2.2 Прикладной слой (internal/application/service/)

Содержит сценарии использования (use cases), разделённые на команды (изменение состояния)
и запросы (чтение). Каждый домен имеет свой сервис:

    crawl_job_service.go  — создание, запуск, отмена, список заданий, управление конфигами
    crawl_task_service.go — выдача задач воркерам, обновление статуса, регистрация результатов
    auth_service.go       — регистрация, вход, обновление токенов
    user_service.go       — управление пользователями
    preview_service.go    — генерация превью страниц
    queue/service.go      — CRUD очередей, назначение очередей на задания

Моки для тестирования генерируются инструментом minimock и хранятся в application/service/mocks/.

2.3 API-слой (internal/api/)

gRPC-обработчики, реализующие сгенерированные из proto интерфейсы:

    api/crawl_job/        — CrawlJobService: создание заданий, конфигов, управление расписанием
    api/auth/             — AuthService: вход, регистрация, обновление токена
    api/user/             — UserService: список и управление пользователями
    api/preview/          — PreviewService: генерация превью
    api/worker/           — WorkerService: регистрация воркеров, удалённый drain/kill
    api/queue_admin/      — QueueAdminService: управление очередями, назначение весов маршрутизации

Дополнительные HTTP-обработчики (не через grpc-gateway) расположены в
internal/interfaces/http/ и internal/interfaces/grpc/.

2.4 Инфраструктурный слой (internal/infra/)

persistence/postgres/       — репозитории PostgreSQL, снапшоты (DB-структуры),
                              конверторы domain <-> DB, управление транзакциями (TxManager),
                              миграции (Goose)
messaging/                  — интерфейс messaging.Client и три реализации:
                              rabbitmq/ (amqp091-go), kafka/ (kafka-go),
                              memory/ (gRPC in-memory broker)
services/fetcher/           — HTTP-загрузчик (http_fetcher.go) и браузерный загрузчик
                              (browser_fetcher.go на базе chromedp)
services/sanitizer/         — HTML-санитизатор (bluemonday)
services/contentstore/      — хранилище бинарных данных MinIO (minio_store.go)
cache/                      — Redis-клиент, Redis rate limiter, in-memory rate limiter,
                              кэш robots.txt
secrets/                    — файловое хранилище секретов с опросом изменений
                              (file_secrets_store.go); формат JSON, ключ -> карта полей
logger/                     — Zap-логгер с поддержкой OpenSearch в качестве вывода


3. Паттерны и механизмы

Outbox-паттерн. Доменные события (TaskEnqueuedEvent и подобные) записываются в таблицу
outbox_events в одной транзакции с бизнес-данными. Фоновая горутина API-сервера читает
необработанные события (processed_at IS NULL) и публикует их в брокер сообщений.

Маршрутизация очередей. Компонент internal/worker/routing/queue_routing_policy.go
реализует взвешенную маршрутизацию с FNV-хэшированием по идентификатору задания. Для
каждого задания можно назначить несколько очередей с весами; выбор очереди при публикации
задачи определяется этими весами. Конфигурация загружается из базы данных и кэшируется на
короткое время (30 секунд по умолчанию).

Файловое хранилище секретов. Учётные данные для подключения к брокерам сообщений хранятся
в JSON-файле (QUEUE_SECRETS_FILE_PATH), который периодически перечитывается фоновым
опросчиком без перезапуска процесса.

Управление транзакциями. TxManager (internal/infra/persistence/postgres/transaction/)
передаёт активную транзакцию через context.Context. Репозитории автоматически обнаруживают
и используют транзакцию из контекста, что позволяет координировать несколько репозиториев
в одной транзакции без явной передачи объекта транзакции.

Шардирование PostgreSQL. При PG_SHARDING_ENABLED=true сервис принимает несколько DSN
(PG_SHARD_DSNS через запятую). gRPC-перехватчик ShardKeyInterceptor читает ключ шарда из
метаданных запроса и выбирает соответствующее соединение.

Мониторинг воркеров. Каждый воркер-процесс запускает WorkerMonitor, который регистрируется
на API-сервере через WorkerService. API-сервер может дистанционно перевести воркер в режим
drain (прекращение приёма новых задач) или принудительно завершить его.


4. gRPC-перехватчики (internal/interceptor/)

Цепочка перехватчиков при каждом запросе:

    Log          — структурированное логирование запроса и ответа (Zap)
    Validate     — валидация proto-сообщений (protoc-gen-validate)
    ShardKey     — извлечение ключа шарда из метаданных
    JWTAuth      — проверка JWT-токена (HS256, golang-jwt/jwt v5)
    RBAC         — проверка ролей (authorization.go): admin, user


5. Аутентификация и безопасность (internal/auth/)

Аутентификация построена на основе JWT-токенов (алгоритм HS256, библиотека golang-jwt/jwt v5).
Пакет auth содержит модуль генерации и верификации токенов (jwt.go), модуль хэширования
паролей с использованием bcrypt (password.go), модуль управления refresh-токенами (refresh.go),
HTTP-middleware для проверки токена во входящих запросах (middleware.go) и модуль ролевой
авторизации (authorization.go), который сопоставляет каждый gRPC-метод с требуемой ролью.
В системе предусмотрены две роли: admin и user. При первом запуске сервера автоматически
создаётся учётная запись администратора по умолчанию на основе переменных окружения
DEFAULT_USER_EMAIL и DEFAULT_USER_PWD, если такой пользователь ещё не существует в базе данных.


6. Брокеры сообщений (internal/infra/messaging/)

Система поддерживает три реализации брокера сообщений, выбор между которыми осуществляется
через переменную окружения MESSAGING_BROKER. Все три реализации удовлетворяют единому
интерфейсу messaging.Client, определённому в файле client.go, что позволяет переключаться
между брокерами без изменения кода воркеров.

Реализация rabbitmq использует библиотеку amqp091-go и работает с двумя очередями, имена
которых задаются переменными RABBITMQ_CRAWL_QUEUE_NAME и RABBITMQ_PARSING_QUEUE_NAME.
Реализация kafka построена на библиотеке kafka-go и работает с топиками, заданными
переменными KAFKA_CRAWL_TOPIC_NAME и KAFKA_PARSING_TOPIC_NAME, в рамках группы потребителей
KAFKA_CONSUMER_GROUP. Реализация grpc_memory подключается к автономному сервису memory_broker
по адресу MEMORY_BROKER_ADDR и использует JSON-кодек поверх gRPC-соединения.


7. Наблюдаемость (internal/telemetry/)

Для сбора телеметрии используется стандарт OpenTelemetry (go.opentelemetry.io/otel). Пакет
telemetry отвечает за инициализацию провайдеров распределённой трассировки и метрик,
настройку контекста распространения трейсов между сервисами и настройку семплирования.
Трейсы и метрики экспортируются по gRPC в OTLP-коллектор. На уровне gRPC-сервера подключён
перехватчик otelgrpc, который автоматически создаёт спаны для каждого входящего и
исходящего вызова. Конфигурации дашбордов Grafana расположены в директории grafana/.


8. Конфигурация (internal/config/env/)

Конфигурация загружается из dotenv-файла функцией config.Load(path). Каждый тип конфигурации
вынесен в отдельный файл пакета env: параметры PostgreSQL (DSN, шардирование) — в pg.go,
параметры RabbitMQ — в rabbitmq.go, параметры Kafka — в kafka.go, параметры брокера памяти
(адрес, ёмкость очереди) — в memory_broker.go, параметры Redis — в redis.go, параметры
MinIO — в minio.go, выбор реализации брокера — в messaging.go, регион воркера для
маршрутизации — в worker_region.go, параметры файлового хранилища секретов (путь к файлу,
флаг слежения за изменениями, интервал перечитывания) — в secrets.go. API-сервер принимает
флаг --config-path (по умолчанию .env); воркеры — флаг --worker-config-path (по умолчанию
.worker.env).


9. База данных

В качестве СУБД используется PostgreSQL. Миграции управляются инструментом Goose; файлы
миграций расположены в internal/infra/persistence/postgres/migrations/ и пронумерованы в
хронологическом порядке. Основные таблицы схемы: crawl_jobs (задания краулинга),
crawl_job_configs (конфигурации заданий с правилами обхода и ExtractionSpec), crawl_tasks
(отдельные задачи — один URL соответствует одной задаче), page_snapshots (ключи S3-объектов
со снапшотами HTML-страниц), extracted_records (извлечённые структурированные данные),
outbox_events (события, ожидающие публикации в брокер), previews (превью страниц),
users (пользователи системы), refresh_tokens (refresh-токены авторизации) и queue_endpoints
(управляемые администратором очереди). Все идентификаторы являются UUID. Внешние ключи
используют ON DELETE CASCADE. Признаком необработанного outbox-события служит значение
NULL в поле processed_at.


10. Административный веб-интерфейс (ui/)

В составе проекта имеется административный веб-интерфейс, реализованный на фреймворке Angular.
Интерфейс содержит следующие разделы: список заданий краулинга с возможностью управления ими,
форму создания нового задания, страницу с деталями задания (прогресс выполнения, ссылки на
файлы экспорта), раздел управления очередями и весами маршрутизации, раздел мониторинга
подключённых воркеров, раздел управления пользователями системы и экран входа.


11. Развёртывание (deploy/helm/distributed-crawler/)

Для развёртывания на Kubernetes предусмотрен Helm-чарт, содержащий шаблоны для всех
компонентов системы. Базовые значения параметров задаются в файле values.yaml.
Файл values-dev.yaml содержит переопределения для среды разработки и включает субчарты с
зависимой инфраструктурой (PostgreSQL, RabbitMQ, MinIO, Redis). Файл values-external-infra.yaml
используется в случаях, когда инфраструктура управляется отдельным релизом. Файл values-prod.yaml
содержит переопределения для production-окружения. Нечувствительные переменные окружения
хранятся в ресурсе ConfigMap, пароли и строки подключения (DSN, URL) — в ресурсе Secret.
Все Pod'ы загружают оба ресурса через механизм envFrom.
